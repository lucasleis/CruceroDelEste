This report covers all seven areas you requested. Every finding is tagged with its source. Where the official documentation is silent or ambiguous, that is stated explicitly.

## 1. Creating a Payment / Preference

### Important architectural note first

The Checkout API (Transparente) uses **two different flows**, and they use different endpoints. Audit which one your backend actually calls:

- **Checkout Pro flow** (redirect-based): creates a *preference* via `POST /checkout/preferences`, receives an `init_point` URL, and redirects the user to MP's hosted page.
- **Checkout API / Transparente flow** (custom UI, card tokenization): creates a *payment* directly via `POST /v1/payments`, using a card token generated by MercadoPago.js on the frontend. **This flow does not use `/checkout/preferences` for the payment itself** — preferences in this context are optional metadata.

The questions below address both since an integration may use preferences for configuration + `/v1/payments` for processing.

---

### Required fields for `POST /v1/payments` (Checkout API direct payment)

The required fields are: `token`, `transaction_amount`, `payment_method_id`, and `payer.email`. `installments` is also required.

The API reference confirms that each of these fields returns a specific error if absent: `Token attribute can't be null`, `Payment_method_id attribute can't be null`, `Transaction_amount attribute can't be null`, `Installments attribute can't be null`.

Additionally, the `X-Idempotency-Key` header is requested in order to safely retry requests without the risk of duplicate payments.

### Required fields for `POST /checkout/preferences` (Checkout Pro)

A preference object requires at minimum `items[]` with each item having a `title`, `quantity`, and `unit_price`. All other fields are optional.

---

### `notification_url`

The `notification_url` must be a valid URL starting with `https://` and must be shorter than 500 characters. The API returns specific errors if either constraint is violated: `"Notification_url attribute must be a valid URL"` and `"Notification_url attribute must be shorter than 500 characters"`.

**Documentation is silent on**: response time SLA requirements. There is no documented maximum response time after which MP considers the endpoint unresponsive. The retry behavior (see Section 2) is the only handling mechanism documented.

---

### `back_urls` and `auto_return`

The `back_urls` attribute defines where the buyer is redirected after payment completion. Three sub-keys exist: `success`, `failure`, and `pending`. The field is optional. `auto_return` must be set to `"approved"` to enable automatic redirection after approved payments with credit cards.

`back_urls` is not required for the Checkout API (Transparente) flow since there is no redirect. It is a Checkout Pro concept.

---

### `external_reference`

The `external_reference` field is used for the seller's own order/reference identifier — seen in preference examples as a string like `"Reference_1234"`.

**Documentation is silent on**: maximum length or character restrictions for `external_reference` in the Checkout API context. The Checkout Pro preference API reference does not document a maximum length for this field. Treat it as a free-form string and apply your own length discipline (keep it under 256 characters to be safe).

---

### Preference expiration

To set expiration, the preference must use `"expires": true` together with `expiration_date_from` and `expiration_date_to` in ISO 8601 format (`yyyy-MM-dd'T'HH:mm:ssz`). If `expires` is not set (or set to `false`), no window is enforced.

**Documentation is silent on**: whether a preference has a built-in default expiry if `expires` is not set. The docs imply it does not expire by default. However, for offline payment methods (boleto, cash), MP notes that payments should have an expiration date set at least 3 days apart to ensure payment is made; if payment occurs after expiration, the amount is refunded to the payer's MP account.

## 2. Webhook Notifications

### Payload format

A Webhook payload for the `payment` topic looks like this:
```json
{
  "id": 12345,
  "live_mode": true,
  "type": "payment",
  "date_created": "2015-03-25T10:04:58.396-04:00",
  "user_id": 44444,
  "api_version": "v1",
  "action": "payment.created",
  "data": { "id": "999999999" }
}
```
The `data.id` field contains the payment ID. The webhook does **not** include the payment status; you must do a `GET /v1/payments/{id}` to retrieve the actual state.

---

### Required HTTP response code

The server must return HTTP 200 (OK) or 201 (Created). If this confirmation is not sent, MP retries.

---

### Retry policy

If a 200 or 201 is not received, MP retries every 15 minutes until a response is received. After the third attempt, the interval is extended, but attempts continue.

A third-party GitHub reference documents the well-known community practice that MP retries for up to 4 days. **The official documentation does not document a maximum retry duration or a maximum retry count.** The statement about 15-minute intervals is the only quantified data in the official docs. Treating "continues indefinitely, intervals grow after 3rd attempt" as the official baseline is the correct reading.

**One critical exception**: fraud alert notifications (`stop_delivery_op_wh` topic) do not follow the usual retry logic. If your server does not respond with HTTP 200 or 201, the notification is lost and will not be resent.

---

### Available notification topics

The full set documented in official sources:

- `payment` — payment created or status changed
- `merchant_orders` — orders created/updated (Checkout Pro)
- `topic_chargebacks_wh` — chargeback created or status changed
- `topic_claims_integration_wh` — claim or dispute initiated
- `stop_delivery_op_wh` — fraud alert (no retry)
- `subscription_preapproval_plan` — subscription plan created/updated
- `subscription_preapproval` — subscription without plan
- `subscription_authorized_payment` — authorized subscription payment
- `automatic-payments` (Card Updater) — card data updated for recurring payments
- `pointintegrationwh` — Point (POS) integration events (only configurable via Your Integrations, not via per-payment notification_url)

Sources: additional info documentation, notification topics documentation.

---

### HMAC signature verification — exact algorithm

MP includes a secret key in all webhook notifications via the `x-signature` header. The header format is: `ts=<timestamp_ms>,v1=<hmac_hex>`. The `ts` value is the notification's timestamp in milliseconds; `v1` is the HMAC-SHA256 hex digest.

The signed message (manifest) is constructed as: `id:<data.id>;request-id:<x-request-id>;ts:<ts>;`. The HMAC is computed using SHA-256 with the application secret key as the key and this manifest string as the message. The resulting hex digest is compared against the `v1` value extracted from the `x-signature` header. Additionally, you can use the timestamp from the header to establish a tolerance margin for delivery delays.

The `x-request-id` header contains a unique request identifier that is part of the signature template — it prevents replay attacks using the same payload but a different request context.

The secret key does not have a documented expiration date. Periodic renewal is not mandatory but is recommended; to rotate it, click the Reset button in Your Integrations > Webhooks.

## 3. Payment Status Lifecycle

### All documented `status` values

The documented statuses are:
- `pending` — user has not completed the payment process
- `approved` — payment credited
- `in_process` — payment is under review
- `rejected` — payment rejected
- `cancelled` — payment canceled after timeout or by either party
- `refunded` — payment refunded
- `charged_back` — chargeback initiated

Also: `authorized` — for two-step capture payments (see below).

The full table of `status`/`status_detail` combinations documented for card payments:

| status | status_detail |
|---|---|

| `in_process` | `pending_contingency` |
| `in_process` | `pending_review_manual` |
| `rejected` | `cc_rejected_bad_filled_card_number` |
| `rejected` | `cc_rejected_bad_filled_date` |
| `rejected` | `cc_rejected_bad_filled_other` |
| `rejected` | `cc_rejected_bad_filled_security_code` |
| `rejected` | `cc_rejected_blacklist` |
| `rejected` | `cc_rejected_call_for_authorize` |
| `rejected` | `cc_rejected_card_disabled` |
| `rejected` | `cc_rejected_card_error` |
| `rejected` | `cc_rejected_duplicated_payment` |
| `rejected` | `cc_rejected_high_risk` |
| `rejected` | `cc_rejected_insufficient_amount` |
| `rejected` | `cc_rejected_invalid_installments` |

For chargebacks:

**Documentation is silent on**: a comprehensive exhaustive list of all `status_detail` values across all payment methods (offline, debit, account money). The table above covers credit cards and chargebacks.

---

### Status transitions — can `approved` revert?

Yes, explicitly. When a chargeback is initiated on an `approved` payment, the status changes to `charged_back` with `status_detail: in_process`. After resolution, it becomes `charged_back/settled` (against seller) or `charged_back/reimbursed` (in favor of seller).

No other `approved` → something-else transition is documented for normal payments. However, this does not mean it cannot happen — it means MP hasn't documented it.

---

### `approved` vs `authorized`

MP supports a two-step payment flow via the `capture` field. When a payment is created with `capture: false`, the amount is reserved but not debited — the payment enters `authorized` status. To complete the payment, a `PUT /v1/payments/{id}` with `"capture": true` moves the payment to `approved / accredited`. Capture is only available for Visa, Cabal, Mastercard, and American Express.

`authorized` means: funds reserved, not yet moved. `approved/accredited` means: funds captured and credited. These are materially different states.

---

### Is money movement guaranteed when `approved` arrives?

The documentation does not make an explicit guarantee about synchronous ledger settlement relative to webhook dispatch. The recommended pattern is to GET the payment status after every webhook, and to treat `approved/accredited` as the signal to fulfill — but no settlement finality guarantee is published.

## 4. Refunds

### Endpoint and method

`POST /v1/payments/{id}/refunds`
Source: API Reference — Create Refund

### Required fields

For a total refund: the request body must be empty (no body required).
For a partial refund: send `{"amount": <partial_amount>}`.
Source: Documentation confirms `amount` is specified in the body for partial refunds.

---

### What does `status: approved` in the refund response mean?

The Create Refund API response includes a `status` field. In the standard (non-Pix) case, a successful response shows `"status": "approved"`, indicating the refund was processed.

This is the terminal success state for card refunds. MP does not document any scenario where a card refund reverts after returning `approved` in the refund resource.

---

### What does `status: in_process` mean?

Refunds can enter `in_process` contingency status — for example, due to communication errors during processing. By default, these errors are surfaced as a `400 Bad Request`, obscuring the actual state. Using the header `X-Render-In-Process-Refunds: true` causes the API to return `201 Created` with `"status": "in_process"` instead, making the state explicit. This header can also be used on `GET` requests for refund lists and specific refunds.

When you receive `in_process`, poll `GET /v1/payments/{id}/refunds/{refund_id}` until the status resolves, or listen for the parent payment's `refunded` webhook.

---

### `X-Idempotency-Key` — required or recommended?

The `X-Idempotency-Key` header is described as being "requested" for payment creation to avoid duplicate payments. It is not stated as strictly required (the API will not reject requests without it), but it is strongly recommended for all write operations to enable safe retries.

The same applies to refunds — the refund API reference shows this header in all examples. Treat it as mandatory in production.

---

### Time limit for refunds

Refunds can be issued within 180 days of the payment approval date.

---

### Partial refunds

Partial refunds are supported. To issue one, send the partial amount in the request body: `{"amount": <partial_amount>}`. Multiple partial refunds can be issued as long as the total refunded does not exceed the original payment amount. Both full and partial refunds are available regardless of payment method.

## 5. Chargebacks

### Topics to subscribe to

You need **at minimum** one of these, depending on what you want to capture:

- `topic_chargebacks_wh` — fires when a chargeback is created or its status changes. Requires activating this event in Your Integrations > Webhooks > Configure Notifications.
- `topic_claims_integration_wh` — fires when a claim or dispute is initiated. Sends notification when a claim or chargeback is initiated.
- `stop_delivery_op_wh` — fraud alert (pre-chargeback). No retry logic — if your server misses it, it's gone.

If you only subscribe to `payments`, you will **not** receive chargeback transition notifications.

---

### Chargeback lifecycle (status and status_detail)


| Payment `status` | Payment `status_detail` | Meaning |
|---|---|---|
| `charged_back` | `in_process` | Chargeback received; dispute in progress |
| `charged_back` | `settled` | Decided against the seller; money withdrawn |
| `charged_back` | `reimbursed` | Decided in favor of seller; money refunded |

Separately, the chargeback object itself has a `coverage_applied` field (null while in process, `true` if seller wins, `false` if seller loses) and `documentation_status` (not_supplied, review_pending, etc.).

---

### What can the seller do?

Upon receiving a chargeback notification, the seller can: (1) query the chargeback via `GET /v1/chargebacks/{id}` or by `payment_id`; (2) if `documentation_required: true` and `date_documentation_deadline` is in the future, upload evidence via `POST /v1/chargebacks/{id}/documentation` (files in .jpg, .png, or .pdf, max 10MB). Uploading documentation changes the chargeback's `documentation_status` to `review_pending`.

---

### Deadline to respond

The documentation_deadline is provided per-chargeback in the `date_documentation_deadline` field of the chargeback object. There is no universal documented deadline — it varies per case. After the deadline, documentation can no longer be submitted.

---

### How chargebacks affect the original payment

When a chargeback is initiated, the associated payment status changes immediately to `charged_back / in_process`. The resolution can take up to 6 months depending on the card brand.

## 6. Security

### Beyond HMAC — other documented security measures

MP's official security recommendations are:
- Always send the `access_token` via the `Authorization: Bearer` header, never as a query parameter.
- Never expose the `access_token` on the frontend or in client-side code.
- Use only `public_key` on the frontend (for tokenization).
- Renew credentials frequently to reduce vulnerability windows.

---

### Credential rotation

To rotate credentials, go to Your Integrations → application → More Options → Renew. When production credentials are renewed, the old credentials remain active for a 12-hour grace period, allowing time to update the integration without downtime.

---

### IP allowlists for webhook senders

**The official MercadoPago documentation does not publish any list of IP ranges or CIDR blocks for webhook sender IPs.** There is no documented allowlist mechanism for webhook traffic. The sole documented authentication method is HMAC signature verification.

---

### Rate limits

**The official MercadoPago Argentina documentation does not publish explicit rate limit figures** (requests per minute, requests per second, or per endpoint). The API returns HTTP 429 when limits are exceeded. Community reports confirm 429 errors exist in practice, but no official thresholds are documented. Implement exponential backoff and handle 429 responses explicitly.

## 7. Testing

### Test accounts and credentials

Test accounts are created via Your Integrations → the application → Test accounts → + Create test account. You must create separate Seller and Buyer test accounts, both from the same country. Test accounts can be assigned a fictional balance for testing account-money payments.

MP provides dedicated test cards per country. The outcome of a test payment is controlled by the **cardholder name** (field `card_holder_name`), not by the card number — a specific name string triggers approved, rejected, in_process, etc. The card number, security code, and expiration date are fixed per card and can be combined with different cardholder names to simulate different scenarios.

The actual card numbers are on the Argentina-specific test card page at `https://www.mercadopago.com.ar/developers/en/docs/checkout-api/integration-test/test-cards` (bot detection prevented direct fetch, but the URL is confirmed active from search results).

---

### Sandbox vs production webhook differences

A dedicated test URL can be configured alongside the production URL in Your Integrations → Webhooks. Testing webhooks must use test credentials. The sandbox environment does support webhook notifications, but only with test credentials and a publicly reachable URL.

**Documentation is silent on**: whether sandbox webhook delivery has different retry behavior or timing characteristics compared to production.

---

### Simulating chargebacks in the sandbox

**The official MercadoPago documentation does not document a way to simulate a chargeback programmatically in the sandbox.** The Webhooks simulator (announced in January 2024) can simulate various webhook types, but the documentation does not explicitly list chargebacks as a simulatable event. This is a gap in the testing tooling that you would need to confirm directly with MP support.
